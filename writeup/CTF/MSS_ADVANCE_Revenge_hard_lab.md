## 1. Executive Summary

**Challenge:** MSS_ADVANCE Revenge
**Category:** Cryptography
**Difficulty:** Hard
**Flag:** `picoCTF{MSS_Advance_but_we_brought_it_back_and_made_it_harder!!!}`

This challenge presented a custom cryptographic scheme based on **Mignotte's Secret Sharing (MSS)** — a polynomial secret sharing system. The flag was used as a seed to generate a degree-29 polynomial with 30 small (256-bit) coefficients over a large (1024-bit) prime field. Twenty evaluation points were published, and the flag was AES-encrypted using the constant term of the polynomial as the key.

The core vulnerability was that the polynomial coefficients were **disproportionately small** relative to the modulus. This allowed a **lattice-based attack** combining:

- **Gaussian Elimination over a finite field (F_p)** to reduce the problem's dimensionality
- **LLL Lattice Reduction** to find a short basis
- **Closest Vector Problem (CVP)** to recover the exact secret coefficients

This writeup will teach you these concepts from the ground up.

## 2. Understanding the Challenge: Code Analysis

> 💡 **Before scanning for vulnerabilities, always read the source code carefully. Every line of `chall.py` is a potential clue.**

### 2.1 Reading the Encryption Script

The first step was downloading and carefully reading `chall.py`. Let's break it down section by section.

**Section 1: Coefficient Generation**
```python
MASTER_KEY = hashlib.sha256(flag).digest()
coeffs = [bytes_to_long(MASTER_KEY)]
for i in range(29):
    co = hashlib.sha256(long_to_bytes(coeffs[-1])).digest()
    coeffs.append(bytes_to_long(co))
```

**What this does:**
- The flag is hashed with SHA-256 to produce a 32-byte (256-bit) `MASTER_KEY`
- This key becomes the first polynomial coefficient `c[0]`
- Each subsequent coefficient `c[i]` is derived by SHA-256 hashing the previous one
- This creates a **deterministic chain** of 30 coefficients, all exactly 256 bits wide

**Why this matters:** SHA-256 always produces a 256-bit output. Since `p` is a 1024-bit prime, all 30 coefficients are **4× smaller** than the modulus. This is the critical weakness.

**Section 2: Polynomial Evaluation**
```python
def eval_poly(x, coeffs):
    total = 0
    for i in range(len(coeffs)):
        total *= x
        total += coeffs[i]
    return total % p
```

**What this does:**
This uses [Horner's method](https://en.wikipedia.org/wiki/Horner%27s_method) to evaluate a polynomial efficiently:

$$f(x) = c_0 x^{29} + c_1 x^{28} + \cdots + c_{28} x + c_{29} \pmod{p}$$

**Section 3: Publishing the Evaluation Points**
```python
pairs = []
for i in range(20):
    x = random.randint(0, p)
    y = eval_poly(x, coeffs)
    pairs.append((x, y))
```

**What this does:** Generates 20 `(x, y)` pairs by evaluating the secret polynomial at 20 random points modulo `p`. These pairs are publicly published in `output.txt`.

**Section 4: Flag Encryption**
```python
def encrypt_flag(flag):
    iv = b"\x00"*16
    cipher = AES.new(MASTER_KEY, AES.MODE_CBC, iv)
    enc = cipher.encrypt(pad(flag, 16))
    return iv.hex(), enc.hex()
```

**What this does:** Encrypts the flag using AES-CBC with `MASTER_KEY = c[0]` and a fixed all-zero IV. If we can recover `c[0]`, we decrypt the flag.

### 2.2 Identifying the Vulnerability: The Size Mismatch

| Parameter | Size |
|---|---|
| Prime `p` | 1024 bits |
| Each coefficient `c[i]` | 256 bits (SHA-256 output) |
| Number of coefficients | 30 |
| Number of known evaluations | 20 |

The key observation is:

> **The coefficients are constrained to be small (256-bit) in a space that is large (1024-bit). This is the Hidden Number Problem — and it is solvable with lattice mathematics.**

If the coefficients were uniformly random 1024-bit numbers, the 20 evaluations would give us exactly enough information to reconstruct them via simple polynomial interpolation. But the adversary thought making the coefficients SHA-256 outputs was safe. It wasn't — it made the problem *easier* to attack.

## 3. The Attack: Lattice Cryptanalysis

> 💡 **This section explains lattice attacks from scratch. Don't skip it — understanding the math is what separates a solver who copies scripts from one who can adapt to novel challenges.**

### 3.1 What is a Lattice?

A **lattice** is a grid of points in multi-dimensional space, generated by taking all integer linear combinations of a set of **basis vectors**.

Imagine a 2D grid where your basis vectors are `v1 = (3, 1)` and `v2 = (1, 2)`. Every point you can reach by computing `a*v1 + b*v2` (where `a`, `b` are integers) is on the lattice.

```
  . . . . .
  . . * . .    * = lattice points
  . * . . *
  . . . * .
  * . . . .
```

Two critical problems in lattice cryptanalysis:

- **Shortest Vector Problem (SVP):** Find the shortest non-zero vector in the lattice
- **Closest Vector Problem (CVP):** Given a target point NOT on the lattice, find the closest lattice point to it

### 3.2 Why Does This Help Us?

Our problem is: we know 20 equations of the form

$$c_0 x_j^{29} + c_1 x_j^{28} + \cdots + c_{29} \equiv y_j \pmod{p}$$

for `j = 0, ..., 19`. We want to find the coefficients `c[i]`, which we know are **small** (bounded by `2^256`).

This is a **system of modular linear equations with small unknowns** — the textbook setup for a lattice attack. The idea is:

> If we construct a lattice whose short vectors correspond to solutions of our equation system, then LLL reduction will find those short vectors — and we get our secret coefficients back.

### 3.3 Step-by-Step Attack Construction

#### Step 1: Gaussian Elimination over F_p

**What we did:**

With 20 equations and 30 unknowns, we used Gaussian elimination modulo `p` to solve for 20 "pivot" coefficients in terms of the remaining 10 "free" coefficients:

```python
# Build A*c = y mod p
A = []
y = []
for xj, yj in pairs:
    row = [pow(xj, n - 1 - i, p) for i in range(n)]
    A.append(row)
    y.append(yj % p)
```

After elimination:
- **Pivot columns:** `[0, 1, 2, ..., 19]` — these 20 coefficients are expressed as functions of the free ones
- **Free columns:** `[20, 21, 22, ..., 29]` — these 10 coefficients are our true unknowns

**Why this was the right first move:** It reduced our problem from 30 unknowns to 10. Instead of a 51×51 lattice (which failed earlier), we now work in a 30-dimensional space — much more tractable for LLL.

**What we avoided:** We initially tried a direct 51×51 lattice embedding without elimination. The LLL returned a norm-21 short vector — this was a **spurious solution** in the kernel of the lattice, not our target. The small norm was a red flag that we were in the wrong lattice formulation.

#### Step 2: Constructing the Lattice

After elimination, we have:

$$c_{\text{pivot}_i} = u_{0,i} - \sum_j R_{ij} \cdot c_{\text{free}_j} \pmod{p}$$

We want all coefficients to be small. We encode this as a lattice:

```python
dim = d + m    # d=10 free cols, m=20 pivot cols → 30x30 lattice
L = IntegerMatrix(dim, dim)

for j in range(d):
    L[j, j] = 1
    for i in range(m):
        L[j, d + i] = lift((-Rfree[i][j]) % p)

for i in range(m):
    L[d + i, d + i] = p
```

**Breaking down this construction:**
- The top-left `d×d` identity encodes the free variables
- The top-right block encodes how free variables determine pivot variables (mod `p`)
- The bottom-right `m×m` block of `p` values allows modular reductions
- Any integer linear combination of rows that represents a valid solution will have small entries

#### Step 3: LLL Reduction

```python
from fpylll import IntegerMatrix, LLL, CVP
LLL.reduction(L, delta=0.999)
```

**What LLL does:**

The **Lenstra-Lenstra-Lovász (LLL) algorithm** takes our lattice basis and transforms it into a new basis of **shorter, more orthogonal vectors** while preserving the lattice structure. Think of it as reorganizing the grid so the basis vectors point more "efficiently."

- `delta=0.999`: The Lovász condition parameter. Closer to 1.0 means better reduction quality but slower runtime. We used 0.999 for maximum quality.

**What it does NOT do:** LLL does not directly give us our answer. It gives us a better basis to then solve CVP.

**Why not BKZ instead?** BKZ (Block Korkine-Zolotarev) is stronger than LLL but slower. For a 30-dimensional lattice with 256-bit targets in a 1024-bit field, LLL was sufficient — and we confirmed this when CVP succeeded.

#### Step 4: Solving CVP (Closest Vector Problem)

```python
target = tuple([0]*d + [(-u) for u in u0])
vec = CVP.closest_vector(L, target)
```

**What this does:**

Our target vector encodes "what the pivot values would be if all free variables were zero." The CVP oracle finds the lattice vector closest to this target — and that closest vector encodes our actual small coefficients.

**The intuition:** Our secret solution is a lattice point. The target is a point near it (but not quite on the lattice). CVP finds the closest lattice point — which is our solution.

**Why this worked:** Because our coefficients are only 256 bits in a 1024-bit space, the solution vector is dramatically shorter than a random lattice vector. The geometry strongly favors finding it.

#### Step 5: Extracting Coefficients and Decrypting

```python
free_vals = diff[:d]
piv_vals = diff[d:]

coeffs = [None] * n
for j, c in enumerate(free_cols):
    coeffs[c] = free_vals[j]
for i, c in enumerate(pivot_cols):
    coeffs[c] = piv_vals[i]
```

We then triple-verified the recovered coefficients:

1. **Polynomial check:** Does `eval_poly(x, coeffs) == y` for all 20 pairs? ✅
2. **SHA-256 chain check:** Does `SHA256(c[i]) == c[i+1]` for all 29 links? ✅
3. **Decryption check:** Does AES-CBC with `c[0]` produce valid plaintext? ✅

```python
key = long_to_bytes(coeffs[0], 32)
pt = AES.new(key, AES.MODE_CBC, b"\x00"*16).decrypt(bytes.fromhex(enc_flag[1]))
```

## 4. The Road Not Taken: What Failed and Why

> 💡 **Understanding failures is just as important as understanding successes. Every CTF player hits dead ends — what matters is recognizing them quickly and pivoting.**

### ❌ Attempt 1: Direct 51×51 Lattice Embedding

**What we tried:**
```
M = 51×51 matrix with p on diagonal and evaluation powers in off-diagonal
```

**Why it seemed reasonable:** This is the "textbook" lattice embedding for hidden number problems. It works well when the number of equations closely matches the number of unknowns.

**Why it failed:** With 30 unknowns and only 20 equations, the lattice had a 10-dimensional kernel of non-trivial short vectors that had nothing to do with our secret. LLL found one of these spurious vectors (with norm 21) instead of our target.

**Red flag we noticed:** A norm-21 vector in a lattice built from 1024-bit numbers is physically impossible as a genuine solution — it was clearly a kernel element.

### ❌ Attempt 2: Pure Python LLL with Float Arithmetic

**What we tried:** Implementing LLL from scratch using Python floats for Gram-Schmidt orthogonalization.

**Why it seemed reasonable:** No external dependencies needed.

**Why it failed:** Python `float` (64-bit IEEE 754) cannot represent 1024-bit integers with any precision. The Gram-Schmidt step divided two enormous integers, causing immediate `OverflowError`.

**Lesson:** For lattice attacks on crypto-sized integers, you **must** use arbitrary-precision arithmetic. `fpylll` internally uses MPFR (arbitrary-precision floating point) to handle this correctly.

### ❌ Attempt 3: SageMathCell Remote API

**What we tried:** Sending our lattice code to `sagecell.sagemath.org` via HTTP POST.

**Why it failed:** HTTP 403 Forbidden — the remote API blocked the request, likely due to rate limiting or network restrictions in the CTF environment.

**Lesson:** Always have a local fallback. Online tools are convenient but unreliable in CTF environments.

### ✅ What Finally Worked: fpylll + Gaussian Elimination + CVP

The winning combination was:
1. Reduce dimensions first (Gaussian Elimination)
2. Use a proper big-integer LLL library (`fpylll`)
3. Frame the problem as CVP rather than SVP

## 6. Lessons Learned & Mitigation

### 🔑 Lesson 1: Small Secrets in Large Fields Are Dangerous

**The vulnerability:** The coefficients were SHA-256 outputs (256-bit) evaluated mod a 1024-bit prime. The 4:1 size ratio made them recoverable via lattice reduction.

**Mitigation:** If building a polynomial secret sharing scheme, ensure coefficients are drawn **uniformly at random** from the full range `[0, p)`, not from a smaller subspace. Never derive secrets from a hash chain — this both limits entropy and creates structural relationships.

### 🔑 Lesson 2: The Number of Shares vs. Degree Tradeoff

**The vulnerability:** A degree-29 polynomial requires 30 shares to reconstruct via Lagrange interpolation. But only 20 shares were published. One might think 20 < 30 means the system is secure. **It isn't** — the small-coefficient structure means lattice attacks succeed with far fewer shares than the polynomial degree.

**Mitigation:** When security relies on a threshold secret sharing scheme, ensure that all polynomial coefficients are full-size random elements of F_p. The threshold `t` only protects against classical interpolation, not lattice attacks on biased coefficient distributions.

### 🔑 Lesson 3: Fixed IVs in AES-CBC Leak Information

**The vulnerability:** The AES-CBC IV was hardcoded as 16 zero bytes. While this didn't affect this specific challenge (we already had the key), fixed IVs in CBC mode are a serious vulnerability in general — they make identical plaintext blocks produce identical ciphertext blocks and enable chosen-plaintext attacks.

**Mitigation:** Always generate a fresh cryptographically random IV for each encryption. The IV is not secret — it should be prepended to the ciphertext and transmitted alongside it.

### 🔑 Lesson 4: Deterministic Key Derivation Reduces Entropy

**The vulnerability:** All 30 coefficients were derived deterministically from a single 256-bit seed via SHA-256. This means the entire 30-coefficient polynomial had only **256 bits of entropy**, not `30 × 256 = 7680` bits as one might naively assume.

**Mitigation:** Generate each coefficient independently using a cryptographically secure random number generator (CSPRNG), not a hash chain.
